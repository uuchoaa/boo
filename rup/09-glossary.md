# Boo — Glossary

**Version:** 1.0  
**Date:** 2026-03-07  
**Status:** Draft  

---

## Terms

**Atomic commit**  
A commit that encodes exactly one intention — one test, one feature, one refactor, or one documentation update. An atomic commit touches 1–2 files and can be described completely by its commit message. Contrast with *fat commit*.

**Benchmark (model benchmark)**  
A structured evaluation of an LLM's output against the fingerprint rubric for a given issue. Results are logged in the benchmark table to detect regressions across model versions or prompt changes.

**Boo**  
The ghost developer system. Boo accepts an issue + codebase context, generates an ordered commit sequence that replicates a developer's fingerprint, and schedules the commits for deployment via git.

**Burst pattern**  
Jordan Casey's characteristic temporal fingerprint: commits arrive in rapid succession (3–8 minutes apart) during an implementation session, followed by a long gap (60–180 minutes) before tests are written.

**Commit density**  
The number of commits generated for a single feature or issue. Rafael's density: 6–10. Jordan's density: 3–6.

**Commit message style**  
The format and tone of commit messages. Rafael uses conventional commits with descriptive 5–8 word descriptions (e.g., `feat: add ghosted scope to Opportunity model`). Jordan uses short, informal messages (e.g., `wip`, `fix`, `almost there`).

**Commit order**  
The temporal sequence of commit types within a feature. Rafael's canonical order: `test → feat → refactor → docs`. This order encodes the developer's workflow philosophy (TDD, incremental refinement).

**Commit type**  
One of: `test`, `feat`, `refactor`, `fix`, `docs`, `ops`, `chore`, `misc`. Derived from the conventional commit prefix in the message.

**Conventional commits**  
A commit message convention using a structured prefix: `<type>: <description>`. Examples: `feat: add user authentication`, `refactor: extract threshold constant`. Enforced by Rafael's fingerprint; ignored by Jordan's.

**Deploy window**  
The time range within a future calendar day during which commits are scheduled. Chosen to overlap with the target company's working hours. Example: `13:00–17:00 BRT` to overlap EST business hours.

**Delta time (delta_minutes)**  
The elapsed time in minutes between two consecutive commits. Injected by the post-processor based on the developer's temporal distribution profile.

**Developer fingerprint**  
The complete stylistic identity of a developer, encoded across 14 code dimensions and 7 writing dimensions. The fingerprint is the primary input to the system prompt (BLOCK 1/2) and is the standard against which generated output is scored.

**Diff (unified diff)**  
The standard format for expressing code changes as a set of added (`+`) and removed (`-`) lines with context. Generated by `git diff` and applied by `git apply`. Boo generates unified diffs as strings within the JSON commit schema.

**Draft PR**  
A GitHub pull request opened in draft state via `gh pr create --draft`. Draft PRs are visible but do not trigger merge requirements or CI status checks. Used by Boo to allow developer review before the PR is marked ready.

**Fat commit**  
A commit that resolves multiple concerns simultaneously — multiple files, multiple intentions. Characteristic of Jordan Casey's style. Contrast with *atomic commit*.

**Fingerprint fidelity**  
The degree to which generated commits authentically replicate the target developer's style. Measured as a weighted score (0–10) across 5 dimensions: commit order (30%), granularity (20%), messages (20%), design decisions (20%), tests (10%).

**Ghost dev (ghost developer)**  
A system that generates code changes indistinguishable from a specific developer's organic output. Boo is a ghost dev system.

**Golden sample**  
A human-generated commit sequence used as the reference standard for benchmark scoring. Rafael Uchôa's output on issue #42 (score: 8.5/10) is the current golden sample.

**`git apply --index`**  
A git command that applies a unified diff to the staging area (index) without modifying the working tree. Used by `deploy.sh` to apply diffs before committing.

**`git commit --date`**  
A git option (`GIT_AUTHOR_DATE` + `GIT_COMMITTER_DATE` environment variables) that overrides the commit timestamp. Used by Boo to backdate commits to their scheduled timestamps.

**Jitter**  
A random offset (±5 minutes) applied to scheduled commit timestamps to avoid perfectly uniform spacing, which would appear artificial in git history.

**Magic number**  
A hardcoded numeric literal in source code without a named constant. Example: `where('updated_at < ?', 7.days.ago)`. Rafael always extracts magic numbers into named constants (e.g., `GHOSTED_FLAG_THRESHOLD = 7.days`). Jordan leaves them hardcoded.

**Normal distribution (commit timing)**  
The inter-commit delay distribution used for Rafael's profile: μ=90 minutes, σ=30 minutes, minimum 20 minutes. Models a developer working at a steady, thoughtful rhythm.

**OpenClaw**  
An open-source local AI agent runtime. Used as the optional execution layer for Boo, enabling cron-based scheduling, persistent state in Markdown files, and natural language workflow triggers.

**Post-processor**  
`post_processor.rb` — the Ruby script that injects timestamps into a commit plan JSON. Supports two modes: past-anchored (for local review) and schedule-windowed (for deploy).

**PR description**  
The body text of a GitHub pull request, generated by Boo using BLOCK 10. Rafael's style: What/Why/How, concise, no emoji, no padding, no sign-off.

**Review response**  
GitHub review comment replies and corresponding fix commits generated by Boo using BLOCK 9. Rafael's style: point-by-point, references code directly, no "great catch", technical pushback when disagreeing.

**Rubric**  
See *scoring rubric*.

**Schedule mode**  
The `--schedule` flag on `post_processor.rb`. Redistributes commit timestamps into a specified future time window. Enforces: date > today, date in the future.

**Scoring rubric**  
The 5-dimension weighted scoring framework used to evaluate fingerprint fidelity: commit order (30%), granularity (20%), commit messages (20%), design decisions (20%), tests (10%).

**Self-evaluation**  
BLOCK 6 — a prompt pasted into the LLM conversation after generation, instructing the model to score its own output against the fingerprint rubric, identify bugs, and list improvements.

**System prompt (BLOCK 1/2)**  
The LLM system prompt that encodes a developer's fingerprint. BLOCK 1 = Rafael Uchôa. BLOCK 2 = Jordan Casey. Defines rules, fingerprint dimensions, pre-generation checklist, and typical commit sequence.

**Temporal fingerprint**  
The timing dimension of a developer's commit behavior: inter-commit intervals, session patterns, and day/time preferences. Encoded in the `delta_distribution` and `session_pattern` fields of `CodeFingerprint`.

**Unified diff**  
See *diff*.

**WIP tolerance**  
Whether a developer commits work-in-progress. Rafael: never. Jordan: without guilt. Encoded in the `wip_tolerance` fingerprint dimension.

**Writing fingerprint**  
The 7-dimension encoding of a developer's written communication style in PR descriptions and review comments. Dimensions: comment length, emoji usage, review response style, pushback style, acknowledgment, PR description style, sign-off.
